# 提示词迭代记录与平台输出规则

## 一、每次迭代必须记录

以后每次用户让我改提示词、测试提示词或反馈出图结果时，都要记录到对应案例文档中，至少包含：

- 日期
- 使用平台：ChatGPT / GPT / nanobanana / nanobanana PRO / 其他
- 使用的提示词版本
- 用户反馈：哪里好、哪里不好、是否可用
- 原因分析：可能是提示词问题、模型设置问题、参考图权重问题、风格词冲突、布局约束不够等
- 下一版调整方向
- 最终沉淀出的可复用规律

## 二、ChatGPT / GPT / nanobanana 不按 SD 方式单独输出负面提示词

Stable Diffusion 常见“正向提示词 + 单独 Negative Prompt”写法，不应默认套用到 ChatGPT / GPT 绘图或 nanobanana。

对于 ChatGPT / GPT / nanobanana，默认输出一个完整的自然语言提示词，把限制条件直接写进正文，例如：

```text
不要改变体型、不要改变三视图结构、不要改变画布比例、不要使用强方向光、不要遗漏胸前和背后的 4 号。
```

只有在用户明确要求“给我负面提示词”、使用的平台明确有 Negative Prompt 输入框，或者为了沉淀规则库需要单独整理禁忌词时，才额外提供单独的负面提示词区块。

## 三、推荐输出格式

默认给用户的可复制提示词应包含：

1. 一段完整提示词：可直接复制到 ChatGPT / GPT / nanobanana。
2. 可选补充说明：如果需要，说明哪些词是关键约束。
3. 不默认拆成 SD 式 Positive Prompt / Negative Prompt。

## 四、GitHub / PR 操作说明

当前环境可以在当前分支修改文件、提交 commit，并创建 PR 记录。是否能直接覆盖 GitHub 默认分支，取决于远程仓库权限、分支保护规则和凭据配置。

为减少用户手动操作，默认流程固定为：

1. 我直接修改仓库文件。
2. 我运行基础检查。
3. 我提交 commit。
4. 我创建 PR 记录。
5. 用户只需要在 GitHub 上 review / merge。

如果用户希望进一步减少手动操作，可以在仓库中配置自动合并或给当前环境配置允许推送 / 合并的凭据；但在没有明确权限和分支保护信息前，不默认强推或覆盖远端分支。

## 五、GitHub 访问失败时的本地优先工作流

如果用户明确说明“当前 Codex 终端无法访问 GitHub，请不要运行 `git fetch` / `git pull` / `git ls-remote`”，则后续默认遵守：

1. 不再尝试从终端访问 GitHub 远端。
2. 直接基于当前工作区已有代码修改文件。
3. 修改后运行本地可用检查，例如 `git diff --check`。
4. 提交本地 commit。
5. 结束时给出本地 diff 摘要、改动文件和测试结果。

这个流程可以避免 GitHub 代理 / 权限问题反复打断提示词迭代，但不能自动解决远端 PR 冲突，也不能保证当前工作区已经包含 GitHub 默认分支的最新内容。
